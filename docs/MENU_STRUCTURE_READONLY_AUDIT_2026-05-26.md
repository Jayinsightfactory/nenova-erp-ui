## 모바일 `/m` 검증 결과

중요 불일치가 확인됐다.

`pages/_app.js`는 `/m/` 하위 경로만 레이아웃 제외 처리하고 있고, 정확히 `/m` 경로는 제외하지 않는다.

현재 코드:

```js
const isNoLayout =
  NO_LAYOUT.includes(router.pathname) || router.pathname.startsWith('/m/');
```

결과:

- 운영 `/m` 접속 시 모바일 홈이 독립 화면으로 뜨지 않고, 데스크톱 `Layout` 사이드바 안에 들어간 상태로 보인다.
- 운영 `/m` 본문에는 `로딩 중...`이 보였다.
- 즉, `/m` 자체는 모바일 메뉴 검증이 정상적으로 되지 않는 상태다.

판정:

- `/m`은 `router.pathname === '/m'`도 레이아웃 제외 대상이어야 한다.
- 현재 상태에서는 모바일 홈 메뉴와 실제 운영 렌더링이 100% 일치한다고 볼 수 없다.

## 데스크톱 메뉴와 모바일 복제 메뉴 비교

`components/Layout.js`의 데스크톱 메뉴와 `pages/m/index.js`의 `DESKTOP_MENU` 복제본을 비교했다.

결과:

- 데스크톱 메뉴 수: 37개
- 모바일 복제 메뉴 수: 38개

모바일 복제본에만 있는 항목:

- `/m/admin/status` - `진단/헬스체크`

라벨/순서 차이:

| 항목 | 데스크톱 | 모바일 복제본 |
| --- | --- | --- |
| `/shipment/stock-status` | 출고,재고상황 | 출고/재고상황 |
| `/shipment/week-pivot` | 차수피벗 | 차수피벗 (전체화면) |
| `/purchase/status` | 구매현황(외화/수입) | 구매현황 (외화/수입) |

순서 차이:

- 데스크톱: `출고,재고상황` 다음 `차수피벗`
- 모바일 복제본: `차수피벗` 다음 `출고/재고상황`

판정:

- 모바일 복제 메뉴는 데스크톱 메뉴와 100% 동일하지 않다.
- 차이가 의도된 모바일 전용 진단 메뉴라면 별도 그룹으로 분리하는 것이 안전하다.
- “데스크톱 전체 메뉴와 동일”이라는 주석을 유지하려면 라벨/순서/항목을 정확히 맞춰야 한다.

- `/m` 자체가 레이아웃 제외에서 빠져 운영에서 데스크톱 사이드바 안에 렌더링되는 문제
- 데스크톱 운영 메뉴는 `407a7f2` 코드 기준으로 정상 반영
- 모바일 복제 메뉴는 데스크톱과 1개 항목 및 일부 라벨/순서가 다름

## 수정 제안

데이터 입력 없이 가능한 안전한 수정 후보:

1. `/m` 레이아웃 제외 조건 보완

```js
const isNoLayout =
  NO_LAYOUT.includes(router.pathname) ||
  router.pathname === '/m' ||
  router.pathname.startsWith('/m/');
```

2. `pages/m/index.js`의 `DESKTOP_MENU`를 `components/Layout.js` 기준과 동기화

- `출고,재고상황` 라벨 통일
- `차수피벗` 라벨 통일
- `구매현황(외화/수입)` 라벨 통일
- 메뉴 순서 통일
- `/m/admin/status`는 모바일 전용 진단 메뉴로 별도 섹션 유지

3. 장기적으로 메뉴 정의를 한 파일로 분리

예:

- `lib/menuConfig.js`
- 데스크톱 Layout과 모바일 메뉴가 같은 source of truth 사용

이렇게 하면 메뉴 불일치 재발 가능성을 줄일 수 있다.

## 결론

현재 운영 데스크톱 메뉴는 코드와 동일하게 렌더링된다.

하지만 전체 페이지 구성 관점에서는 아래 때문에 100% 동일/정상이라고 판정할 수 없다.

- `/m` 경로 레이아웃 제외 누락
- 모바일 복제 메뉴와 데스크톱 메뉴 간 항목 수/라벨/순서 차이
- 팝업 메뉴는 DOM에서 `href="#"`로 표시되어 실제 target URL은 코드 기준 검증이 필요

이번 검증 중 데이터 입력, 저장, 수정, 삭제 작업은 하지 않았다.
