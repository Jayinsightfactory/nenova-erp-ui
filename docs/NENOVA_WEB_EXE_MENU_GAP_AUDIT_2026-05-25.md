# NenovaWeb vs Nenova.exe 메뉴/화면 누락 감사 - 2026-05-25

## 목적

로그인 후 첫 화면부터 NenovaWeb의 전체 메뉴, 버튼, 화면 정보가 기존 `nenova.exe`와 어디까지 맞고 어디가 다른지 재확인했다.

이번 문서는 새로 만든 웹 전용 기능은 제외하고, 기존 exe 업무 흐름을 웹으로 옮기는 관점에서 누락 또는 차이 가능성이 큰 항목을 우선순위로 남긴다.

## 확인 기준

- 웹 메뉴: `components/Layout.js`, `pages/dashboard.js`, `pages/m/index.js`
- 기존 문서: `WEB_VS_ERP_CONFLICTS.md`, `FULL_FLOW_AUDIT_2026-05-15.md`, `UI_FLOW_AUDIT_2026-05-15.md`, `CHATBOT_LEARNING_AUDIT_2026-05-25.md`
- exe 기준: dnSpy/문자열 분석으로 확인된 `FormOrderAdd`, `FormShipmentDistribution`, `FormWarehouseAdd`, `FormEstimateView`, `FormStockView`, `FormCustomerInfo`, `FormCustomerProdCost`, `FormHistory`, 통계/분석 폼

## 이전 대화 기준 누락 작업 재확인

1. 출고분배는 아직 최우선이다. 일부 품목만 분배할 때 다른 품목의 확정 상태 때문에 막히면 안 된다. 예: 카네이션 매칭 후 분배라면 카네이션 범위만 확정 상태를 봐야 한다.
2. 매칭 히스토리/학습은 실제 주문 저장 및 출고분배 성공 이후에만 반영되어야 한다. 매칭 후보를 고르는 시도만으로 자동 매핑이 바뀌면 안 된다.
3. 출고분배 시 단가는 `CustomerProdCost` 기준으로 자동 입력되어야 하고, 이 값이 견적서 관리에 이어져야 한다.
4. 출고일이 지정되지 않은 상태에서 분배되는 흐름은 재검증해야 한다. exe의 `ShipmentDate` 합계 검증과 웹 저장 흐름이 같아야 한다.
5. `ShipmentFarm` 보조 테이블 갱신 정책이 아직 100% 확정되지 않았다. 견적서 출력/농장 분배와 관련해 exe와 차이가 날 수 있다.
6. `StockList` view 정의와 `Product.Stock`/`ProductStock` 동시 운용 문제는 계속 미해결 위험이다.
7. Ecount/경영지원 화면은 웹 계산값 중심이라 ERP 원본 판매/채권/세금계산서/구매 원장과 아직 100% 일치 검증 전이다.
8. 챗봇은 모델 학습이 아니라 런타임 메모리와 감사 로그 구조다. `_chat_audit`로 답변 품질을 검토하고, 반복 오답은 고정 handler로 승격하는 방식이 필요하다.
9. 저장된 자동 매핑 중 과거 fallback 오매칭 후보 정리가 남아 있다.
   - 중국 장미류가 주밀리아로 몰린 케이스
   - generic HYDRANGEA 수국류
   - ALSTROMERIA Arrone 몰림
   - 레몬잎/태국/호접류가 폼폼이 화이트로 몰린 케이스
10. 운영 Product 전체 목록 기준 자연어 매칭 사각지대 재검증과 실제 카톡 문장 샘플 테스트가 남아 있다.

## 작업 전 충돌 방지 지시

위 항목을 수정하기 전에는 `PRE_WORK_CONFLICT_CHECK_2026-05-25.md`를 먼저 확인한다.

특히 주문, 출고, 견적, 재고, 정산, 챗봇 DB 쓰기 작업은 아래 순서를 지킨다.

1. exe 버튼/폼/SP 기준 확인
2. 운영 DB 기존 row와 KeyNumbering 상태 확인
3. 웹 코드가 생성/수정하는 테이블 확인
4. 충돌 없는 `+A` 보완안 설계
5. 코드 수정
6. build 확인
7. 운영 git-log 및 실제 화면 확인
